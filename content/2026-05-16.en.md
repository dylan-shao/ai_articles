## Copilot Memory supports user preferences for Pro, Pro+ users - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-05-15
- URL: https://github.blog/changelog/2026-05-15-copilot-memory-supports-user-preferences-for-pro-pro-users/

GitHub expanded Copilot Memory from repo-level memory to user-level preference memory. The system can store explicitly stated or inferred personal preferences and reuse them across future interactions, repositories, and Copilot experiences, including commit style, PR structure, and communication tone.

Why it matters: This matters beyond simple personalization: it means coding agents are beginning to build durable, cross-context models of how an individual works instead of relearning preferences every session. For engineering orgs, this has direct implications for prompt design, review consistency, agent permission boundaries, privacy governance, and how team standards coexist with personal preferences.

Key takeaways:
- The memory layer is expanding from repository context to user profile, indicating a shift toward durable individual adaptation.
- Users can review and delete personal memories, showing that controllability and privacy governance are becoming first-class product concerns.

Tags: copilot, agent-memory, developer-tooling, personalization

## Grok Code Fast 1 deprecated - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-05-15
- URL: https://github.blog/changelog/2026-05-15-grok-code-fast-1-deprecated/

GitHub has deprecated Grok Code Fast 1 across all Copilot experiences and recommends moving to GPT-5 mini or Claude Haiku 4.5. Enterprise admins may need to enable the replacement models via policy, and workflows/integrations should be updated accordingly.

Why it matters: This is more than a simple model retirement notice: it highlights that the model supply layer behind AI coding platforms is fluid. If your IDE, CLI, code review, or agent workflows depend on specific models, you need governance for model substitution, compatibility testing, and admin policy changes.

Key takeaways:
- Foundation models in the coding-agent stack are not stable dependencies; enterprises should treat model replacement as a formal change-management concern.
- Admin model policy is now a key control plane for both availability and developer experience.

Tags: copilot, model-governance, enterprise-adoption, change-management
Scores: signal 6/5, novelty 5/5, actionability 8/5

## Timestamp fields in GitHub Projects - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-05-15
- URL: https://github.blog/changelog/2026-05-15-timestamp-fields-in-github-projects/

GitHub Projects now includes built-in timestamp fields for Created, Updated, and Closed, enabling sorting and filtering across issues, draft issues, and pull requests. The Updated field also reflects project field changes, so the same item can show different timestamps across projects.

Why it matters: The feature is modest on its own, but useful for platform and engineering productivity teams: these fields make it easier to build views like 'recently completed,' 'stale work,' or 'recent status changes,' and provide cleaner data for agent-driven project automation and workflow metrics.

Key takeaways:
- Projects is becoming a more automation-friendly workflow data surface, not just a manual board.
- Because Updated includes project field changes, it is useful for tracking process movement rather than only code-object activity.

Tags: github-projects, workflow-automation, engineering-metrics
Scores: signal 5/5, novelty 4/5, actionability 6/5
