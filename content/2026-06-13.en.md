## Copilot code review: New configurations and controls - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-06-12
- URL: https://github.blog/changelog/2026-06-12-copilot-code-review-new-configurations-and-controls/

GitHub added three important enterprise controls to Copilot code review: organization-level runner type configuration, content exclusion support, and removal of the character limit for repository custom instructions. Because Copilot code review uses an agentic architecture powered by GitHub Actions, these changes effectively add centralized governance over execution environment, visible data scope, and behavioral guidance for AI review.

Why it matters: This is more than a minor feature update; it is a clear signal that AI code review is entering a platform-governance phase. For platform teams, the hard problem is no longer getting an agent to run, but scaling it under compliance, isolation, performance, cost, and org-wide consistency constraints. Org-level runner locking lets enterprises route AI review workloads onto self-hosted or larger runners by default; content exclusion enables practical handling of sensitive code and policy boundaries; and longer custom instructions allow teams to encode review norms, architectural constraints, and style rules as operational policy.

Key takeaways:
- Copilot code review is now tightly coupled to the GitHub Actions execution plane, making runner selection part of AI review architecture.
- Org-level defaults and locking show the adoption focus shifting from giving developers an AI feature to defining controlled behavior for the whole organization.
- Content exclusion and larger instruction budgets make AI review look more like a governable internal platform capability than a point feature.

## GitHub Actions: Minimum version enforcement timeline for self-hosted runners - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-06-12
- URL: https://github.blog/changelog/2026-06-12-github-actions-minimum-version-enforcement-timeline-for-self-hosted-runners/

GitHub is resuming minimum version enforcement for self-hosted Actions runners and provided a timeline plus compatibility requirements. The post also reveals the infrastructure context: the new Actions architecture now handles more than 120 million jobs per day and materially improves enterprise job-start throughput. As older runners become incompatible with the new platform, upgrading runner versions becomes not just a CI maintenance task but a direct dependency for Copilot code review and cloud agents that rely on Actions.

Why it matters: For platform leaders, the key value here is that it ties AI developer-tool availability to CI/CD infrastructure hygiene. If GitHub is placing more agentic capabilities on the Actions execution plane, then runner lifecycle management, version drift, network egress, isolation levels, and auto-upgrade policy all become gating factors for AI adoption. Many teams will buy the AI capability first and only later discover that the real blocker is an aging runner fleet and weak platform governance.

Key takeaways:
- Self-hosted runner version management is evolving from a reliability concern into an AI workflow availability concern.
- GitHub is signaling that agentic developer experiences will increasingly be built on top of large-scale Actions infrastructure.
- Platform teams should audit runner versions, registration flows, and upgrade automation now to avoid AI review or agent execution being blocked by infra prerequisites.
