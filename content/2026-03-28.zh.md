## Quantifying infrastructure noise in agentic coding evals
- Source: Anthropic Engineering
- Author: N/A
- Publish date: N/A
- URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic 讨论了 agentic coding benchmark 中常被忽略的一层变量：基础设施配置本身。文章指出，在 Terminal-Bench 2.0 的内部实验中，不同资源配置与执行方式可造成约 6 个百分点的成绩差异，已经超过许多前沿模型在榜单上的分差。由于这类评测要求模型在真实运行环境中写代码、跑测试、安装依赖并多轮迭代，运行时不再是被动容器，而是评测的一部分，因此 CPU、RAM、时限和资源 enforcement 方式都会改变“被测的到底是什么”。

Why it matters: 这篇文章直接挑战了团队对 coding benchmark 分数的默认解读方式。对于在做模型选型、代理框架对比、CI 中自动评测或内部 coding agent 竞标的团队来说，如果没有统一且可复现的 harness、资源预算和失败归因机制，那么结论可能反映的是“谁更吃资源、谁更适应某种容器策略”，而不是谁真正更擅长软件工程任务。

Key takeaways:
- Agentic coding eval 与静态 benchmark 不同，运行环境是任务的一部分，因此 infra 不是背景变量，而是核心实验条件。
- 资源预算与 enforcement 方法会显著影响得分；若不统一，模型之间的细微分差缺乏可比性。
- 评测设计应明确 CPU、RAM、超时、重试、容器隔离和 infra error 归因，否则 leaderboard 容易误导采购与部署决策。

## Themes

- AI coding 评测正在从“模型打分”转向“模型+运行环境+执行策略”的系统评测。
- 榜单可比性的瓶颈开始从 prompt 与模型本身，转移到 harness 标准化与资源治理。
- 工程组织需要把 eval 平台当成生产级基础设施来建设。 

## Implications

- 如果你在做内部 coding-agent bake-off，应冻结容器镜像、CPU/RAM 配额、timeout 与重试规则，并单独统计 infra failure。
- 如果你依赖外部 leaderboard 做模型采购，应该追问运行环境与执行约束，否则分数差异可能不可操作。
- 如果你在设计自家 benchmark，应把资源规格和 enforcement 机制写入协议，并提供可复现实验脚本。
