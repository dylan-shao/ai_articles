## Quantifying infrastructure noise in agentic coding evals
- Source: Anthropic Engineering
- Author: N/A
- Publish date: N/A
- URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic shows that in agentic coding evals such as SWE-bench and Terminal-Bench, factors like CPU, RAM, time budget, container scheduling, and the exact resource-enforcement method can materially shift benchmark scores. In their internal Terminal-Bench 2.0 experiments, the gap between the best- and worst-resourced setups reached 6 percentage points—larger than many leaderboard margins between top models.

Why it matters: This is not a generic 'benchmarks are noisy' take; it elevates infrastructure to a first-class experimental variable. For teams comparing models, building internal coding evals, or using benchmark scores for procurement and rollout decisions, the key message is that without tightly standardized and reproducible runtime conditions, you may be measuring harness/platform engineering quality more than model capability.

Key takeaways:
- Unlike static benchmarks, agentic coding evals embed the runtime environment into the task itself, so differences in resource budgets effectively change the exam.
- Publishing recommended resource specs is not enough; the way limits are enforced, containers are scheduled, and timeouts are configured changes what the benchmark actually measures.
- For internal eval systems, infra config, scheduling policy, failure taxonomy, and error budgets must be captured as eval metadata, or results will not be comparable across teams or over time.

## GitHub Copilot CLI now supports Copilot auto model selection
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-04-17T00:00:00+00:00
- URL: https://github.blog/changelog/2026-04-17-github-copilot-cli-now-supports-copilot-auto-model-selection/

GitHub has made auto model selection generally available in Copilot CLI. The system dynamically routes across models such as GPT-5.4, GPT-5.3-Codex, Sonnet 4.6, and Haiku 4.5 based on plan, policy, and rate-limit conditions, while exposing the selected model in the CLI.

Why it matters: This is worth noting but is still a relatively shallow product update: CLI agent tooling is shifting from explicit model choice toward policy-driven auto-routing. For enterprise platform teams, the real question is not the feature itself but whether observability, cost attribution, and policy controls are granular enough.

Key takeaways:
- CLI agents are adopting the same automatic routing layer seen in chat products, making model selection increasingly a platform-policy concern rather than a user setting.
- The announcement mostly covers availability and billing discounts, with little technical detail on routing logic, task adaptation, or quality evaluation.

## Themes

- AI coding evaluation is shifting from pure model comparison to system-level comparison of model plus runtime plus harness.
- Platform and infrastructure details are becoming primary determinants of benchmark reproducibility.
- Developer tools are moving toward default auto-routing across models, but technical transparency remains limited.

## Implications

- If your organization uses external leaderboards for model procurement or rollout decisions, require full disclosure of runtime configs, failure taxonomy, resource limits, and reproduction scripts.
- Internal AI coding eval platforms should log infra versions, container specs, scheduling behavior, timeout policy, and retry policy, while separating model failures from infrastructure failures.
- For CLI/IDE agent products, the next review focus should be cost variance, explainability, and auditability under automatic routing—not just whether more models are supported.
