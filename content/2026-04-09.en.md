## Quantifying infrastructure noise in agentic coding evals
- Source: Anthropic Engineering
- Author: N/A
- Publish date: N/A
- URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic examines an underappreciated variable in agentic coding benchmarks: infrastructure configuration. In internal Terminal-Bench 2.0 experiments, differences in CPU/RAM/time limits and enforcement methodology alone produced a 6-point score swing, larger than many leaderboard gaps between models. The key argument is that in agentic coding evals, the runtime is not a neutral container but part of the task itself, so agents with different resource budgets are effectively not taking the same test.

Why it matters: This matters to any team using SWE-bench, Terminal-Bench, or internal coding harnesses for model selection. It implies benchmark scores cannot be interpreted independently of infra setup; the eval system itself needs versioning, resource normalization, and error analysis, or organizations risk making deployment decisions based on environment variance rather than model capability.

Key takeaways:
- In agentic coding evals, resource budgets and execution limits materially affect scores, sometimes more than model differences do.
- Eval builders must not only specify recommended resources but also define and consistently enforce the enforcement mechanism, or the benchmark drifts in what it measures.
- For internal enterprise evals, infra noise should be treated as a first-class metric, with explicit tracking of infra failure rate, timeouts, retry policy, and scheduling differences.

## Copilot-reviewed pull request merge metrics now in the usage metrics API
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-04-08T00:00:00+00:00
- URL: https://github.blog/changelog/2026-04-08-copilot-reviewed-pull-request-merge-metrics-now-in-the-usage-metrics-api/

GitHub added two organization-level metrics to the Copilot usage metrics API for code review workflows: the total number of PRs that were both reviewed by Copilot and merged, and the median time from PR creation to merge for those PRs. This extends earlier metrics focused on Copilot-authored PRs and starts measuring how Copilot-assisted review affects delivery flow.

Why it matters: This is foundational for organizational AI coding adoption, not just a minor product update. Many enterprises are stuck with usage but weak proof of value; these API metrics let platform teams integrate Copilot review into DORA/engineering effectiveness dashboards for adoption, cycle-time, and workflow bottleneck analysis.

Key takeaways:
- Governance of AI coding is moving beyond seat counts and active-user metrics toward PR lifecycle outcome metrics.
- Platform teams can compare merge speed for Copilot-reviewed PRs against overall baselines to test whether automated review actually shortens delivery time.
- This data is well suited for team-by-team rollout analysis, A/B comparisons, and internal ROI reviews.

Tags: github-copilot, metrics, organizational-adoption, developer-productivity, code-review

## GitHub Copilot in Visual Studio Code, March Releases
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-04-08T00:00:00+00:00
- URL: https://github.blog/changelog/2026-04-08-github-copilot-in-visual-studio-code-march-releases/

GitHub summarized a significant set of Copilot agent updates in VS Code: public-preview Autopilot, per-session permission modes (Default/Bypass Approvals/Autopilot), configurable thinking effort, integrated browser debugging, nested subagents, and a unified editor for chat customization. Taken together, these changes show AI coding evolving from single-turn assistance into configurable, debuggable, multi-step agent systems.

Why it matters: For engineering leaders, the important part is not feature count but the emergence of a governance model: agents now have permission modes, autonomy levels, debugging loops, and task decomposition. Teams need to define which repos/tasks can use Autopilot, which require human approval, and how to observe reliability and cost across subagent chains.

Key takeaways:
- Agent execution control is becoming standardized; permission and autonomy modes will become default governance surfaces in enterprises.
- Integrated debugging and subagents indicate AI coding is increasingly covering complex end-to-end fix workflows, not just code generation snippets.
- Internal platforms should evolve in parallel with policies for approval thresholds, sandbox boundaries, cost budgets, and failure replay.

## GitHub Mobile: Research and code with Copilot cloud agent anywhere
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-04-08T00:00:00+00:00
- URL: https://github.blog/changelog/2026-04-08-github-mobile-research-and-code-with-copilot-cloud-agent-anywhere/

GitHub expanded Copilot cloud agent on mobile beyond PR workflows into broader repo research, implementation planning, branch-based code changes, and diff iteration. Users can now ask the agent to first understand the codebase, propose a plan, and modify a branch before deciding whether to open a PR.

Why it matters: This is not just 'coding on mobile'; the real signal is that cloud agent workflows are decoupling from the IDE and becoming asynchronous execution surfaces available from anywhere. For platform teams, that means agents increasingly resemble remote execution planes rather than local editor plugins, with corresponding needs for identity, audit, and execution boundary design.

Key takeaways:
- AI coding interfaces are expanding beyond desktop IDEs, while execution is becoming more centralized in cloud agents.
- The 'research codebase, plan, then edit branch' workflow shows repo understanding becoming an explicit product capability rather than a hidden prerequisite.
- Enterprises should pay attention to audit, approval, and secrets risks introduced by mobile or remote agent triggering.
