## Quantifying infrastructure noise in agentic coding evals
- Source: Anthropic Engineering
- Author: N/A
- Publish date: N/A
- URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic Engineering analyzes “infrastructure noise” in agentic coding benchmarks such as SWE-bench and Terminal-Bench, arguing that the runtime environment is not a neutral container but part of the test itself. In internal Terminal-Bench 2.0 experiments, resource configuration alone shifted scores by about 6 percentage points—larger than the gap between some top leaderboard entries.

Why it matters: This piece directly challenges how the industry interprets coding-agent leaderboards: if teams run with different CPU/RAM budgets, timeout policies, scheduling, and isolation behavior, they are not taking the same test. For engineering leaders responsible for model evaluation, procurement, platform governance, and internal benchmark design, this affects comparability, fairness, regression detection, and budget planning.

Key takeaways:
- Unlike static benchmarks, agentic coding evals are shaped by the runtime environment, so infra is not a secondary variable.
- Even if a benchmark specifies recommended resources, without consistent and auditable enforcement the result may measure environment advantage rather than model capability.
- Eval platforms should log and publish resource budgets, timeouts, failure modes, scheduling policy, and retry behavior; otherwise leaderboard comparability is weak.

## New Terms

- Infrastructure noise: benchmark variance introduced by resource configuration, scheduling, timeout, and isolation behavior.
- Resource enforcement consistency: the ability to apply CPU, memory, and time constraints uniformly and auditably across runs.

## Themes

- Agentic coding evaluation is shifting from pure model-output scoring to joint evaluation of model plus execution environment.
- Engineering details of the eval harness are becoming central to leaderboard credibility.

## Implications

- Internal eval systems should version benchmark metadata for container shape, concurrency, caching, network access, install policy, and timeout configuration.
- When comparing vendor models or agent frameworks, teams should normalize infra first; otherwise platform tuning can be mistaken for model progress.
- The next quality frontier for benchmarks will be reproducible harnesses, standardized resource budgets, and failure taxonomies—not just better task sets.

## What to Ignore

- Do not read leaderboard percentage gaps without accounting for runtime environment differences.
- Do not assume that “recommended resources” in benchmark docs means strong enforcement and reproducible execution.
