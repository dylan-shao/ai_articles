## Quantifying infrastructure noise in agentic coding evals
- Source: Anthropic Engineering
- Author: N/A
- Publish date: N/A
- URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic argues that in agentic coding benchmarks such as SWE-bench and Terminal-Bench, runtime infrastructure is not a neutral container but part of the task itself. Their internal experiments show that differences in CPU, RAM, time budgets, and resource enforcement can produce a statistically significant 6-point swing on Terminal-Bench 2.0—larger than many leaderboard gaps between models.

Why it matters: This matters because it challenges a common assumption: benchmark scores are often treated as precise proxies for model capability. If resource allocation, scheduling behavior, and timeout policies are not tightly standardized, teams may actually be comparing a 'model × harness × infrastructure configuration' bundle rather than the model itself.

Key takeaways:
- Unlike static benchmarks, agentic coding evals incorporate the runtime environment into the problem-solving loop, so environment differences can systematically alter outcomes.
- Even when a benchmark specifies recommended resources, inconsistent enforcement can change what capability is actually being measured.
- For model selection, internal comparisons, and regression tracking, infra configuration, resource budgets, and infra error rates should be treated as first-class evaluation metadata.

## New Terms

- Infrastructure noise: score variance caused by environmental factors such as CPU, memory, timeouts, scheduling, and container limits.
- Resource enforcement methodology: not just declaring resource budgets, but how those limits are actually imposed in the harness or cluster.

## Themes

- The evaluation environment is becoming part of the measured agent capability rather than a background condition.
- Benchmarking is shifting from pure model comparison to comparison of the full 'model + harness + infrastructure' system.

## Implications

- If an enterprise cannot reproduce a public benchmark, the first suspicion should not be model quality alone; it should also inspect resource allocation, timeout policy, and infra error rates.
- More trustworthy coding evals will likely need standardized execution configs, infra observability metrics, and reproducible harness scripts—not just final scores.

## What to Ignore

- Do not interpret small leaderboard differences as pure model capability differences, especially when gaps are only a few percentage points.
- Do not ignore benchmark harness and cluster execution details; these implementation choices may matter more than prompt tweaks.
