## Selected Items

## Copilot Memory has more controls for deletion, scope, and the Copilot CLI - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-05-26
- URL: https://github.blog/changelog/2026-05-26-copilot-memory-has-more-controls-for-deletion-scope-and-the-copilot-cli/

GitHub added clearer deletion guidance, a repository-level off switch, and CLI support via /memory on|off|show for Copilot Memory. It also makes memory scope explicit at capture time, distinguishing user-level preferences from repository-level facts.

Why it matters: This is more than UX polish; it is governance for the persistence layer behind AI coding. For teams worried about cross-repo leakage, stale/incorrect memory, and opaque long-lived context, these controls materially affect whether memory can be enabled at scale.

Key takeaways:
- Repository admins can now disable Copilot Memory at the repo level, preventing new repo-level facts from being stored or read.
- CLI-level memory control shows GitHub is extending AI behavior configuration beyond the web UI into local developer workflows and automation surfaces.

Tags: copilot-memory, ai-governance, developer-tooling, repo-policy

## Target Copilot models to organizations with model rules - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-05-26
- URL: https://github.blog/changelog/2026-05-26-target-copilot-models-to-organizations-with-model-rules/

GitHub now lets enterprise owners control Copilot model availability per organization instead of relying only on enterprise-wide defaults. Targeted model rules allow different orgs to use different model sets, with a refreshed interface for managing default model availability.

Why it matters: This is a clear signal that AI coding platforms are entering a multi-model governance phase. Large enterprises often need different models by business sensitivity, cost profile, region, team maturity, or eval results; this capability turns model choice into an administrable policy surface.

Key takeaways:
- Model availability is moving from one enterprise-wide setting to org-level segmented policy, which better matches real enterprise structures.
- This creates a practical rollout path for internal evals, phased deployments, cost control, and compliance-based segmentation.

Tags: copilot, model-governance, enterprise-adoption, policy-control
Scores: signal 9/5, novelty 7/5, actionability 9/5

## GitHub Code Quality: Repository Enablement API - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-05-26
- URL: https://github.blog/changelog/2026-05-26-github-code-quality-repository-enablement-api/

GitHub released a public preview Repository Enablement API for Code Quality. The new PATCH/GET endpoints allow repo-level enable/disable, language selection, runner type configuration, and retrieval of current setup and schedule.

Why it matters: Programmable enablement is a key step from feature to platform capability. It means code quality can be rolled out in bulk, integrated with repo bootstrapping/templates, and managed through platform-team automation or IaC-style governance.

Key takeaways:
- GitHub is API-enabling the code quality control plane, which is especially important for fleet-scale repo governance and standardized onboarding.
- Support for languages, runner type, and config retrieval means this is more than a toggle API; it is a manageable and observable interface.

Tags: code-quality, api, platform-engineering, automation
Scores: signal 8/5, novelty 7/5, actionability 9/5

## Code coverage on pull requests is now in public preview - GitHub Changelog
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-05-26
- URL: https://github.blog/changelog/2026-05-26-code-coverage-in-pull-requests-is-now-in-public-preview/

GitHub Code Quality now shows aggregate code coverage percentages directly in pull requests. Teams can upload Cobertura reports to bring coverage signals into PR review, using the new fine-grained code-quality:write permission.

Why it matters: This moves test-completeness signals closer to the merge decision. Coverage itself is not new, but surfacing it natively in PRs and tying it into GitHub Code Quality means quality gates are becoming more embedded in the developer workflow instead of living in separate reporting tools.

Key takeaways:
- Quality signals are shifting from CI byproducts to first-class inputs in the PR decision loop.
- The new fine-grained permission is also a reminder for platform teams to revisit least-privilege settings for Actions and GitHub Apps.

Tags: code-coverage, pull-requests, developer-workflow, quality-gates
Scores: signal 7/5, novelty 6/5, actionability 8/5
