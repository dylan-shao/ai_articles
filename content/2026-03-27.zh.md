## Quantifying infrastructure noise in agentic coding evals
- Source: Anthropic Engineering
- Author: N/A
- Publish date: N/A
- URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic 说明了 agentic coding benchmark 与静态基准的根本差异：运行时环境本身就是能力测量的一部分。其内部实验显示，在 Terminal-Bench 2.0 上，仅基础设施资源与执行配置差异，就能带来 6 个百分点的成绩波动，且显著性达到 p < 0.01。

Why it matters: 这是当前最重要的评测方法论信号之一。很多团队正用 SWE-bench、Terminal-Bench 一类结果来决定模型选型、预算和集成策略；如果 infra 设定本身足以超过模型间 leaderboard 差距，那么你需要把 benchmark harness、资源限额、超时策略和隔离方式视为一等公民，而不是把分数当作可直接比较的事实。

Key takeaways:
- Agentic coding eval 测到的不只是模型，还包括运行环境、资源约束和执行策略。
- 如果不强约束并记录 CPU/RAM/timeout/enforcement 方法，跨团队和跨供应商分数不可直接比较。
- 评测 harness 需要像生产系统一样可复现、可观测、可审计。

Tags: agentic-evals, benchmark-harness, terminal-bench, reproducibility
Scores: signal 10/5, novelty 9/5, actionability 9/5

## Detecting supply chain attacks at scale with Deep Search
- Source: Sourcegraph Blog
- Author: Stephanie Jarmak, Travis Lyons
- Publish date: 2026-03-26
- URL: https://sourcegraph.com/blog/detecting-supply-chain-attacks-at-scale-with-deep-search

Sourcegraph 用 LiteLLM 在 PyPI 的投毒事件做案例，展示如何借助 Deep Search 和 Code Search 快速判定哪些公开仓库受版本 pinning 保护，哪些仓库因未锁版本或使用范围约束而暴露风险，并给出可复用的检索与排查方法。

Why it matters: 这不是普通的产品宣传，而是一个高贴近现实的 repo understanding / code search 运维场景。对平台团队来说，AI 搜索能力的直接价值并不是‘问代码问题’，而是在供应链事件发生时做跨仓库暴露面分析、生成修复清单，并缩短从发现到处置的时间。

Key takeaways:
- 依赖 pinning 与范围约束的差异，在应急时会直接决定暴露面。
- 跨仓库语义搜索适合做快速 triage，而不是只依赖 SBOM 或单仓库 grep。
- 可复现的搜索查询与结果导出流程，应纳入安全事件响应手册。

Tags: repo-understanding, supply-chain-security, code-search, incident-response
Scores: signal 8/5, novelty 7/5, actionability 9/5

## Ask @copilot to resolve merge conflicts on pull requests
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-03-26
- URL: https://github.blog/changelog/2026-03-26-ask-copilot-to-resolve-merge-conflicts-on-pull-requests

GitHub Copilot coding agent 现在可通过在 PR 评论中 @copilot 来自动合并主干并解决冲突。该 agent 在云端开发环境中执行修改、运行构建与测试，然后把结果推回 PR；同一机制也可用于修复 failing workflows、回应 review comments 和补充测试。

Why it matters: 这代表 agent 正从 IDE 内联建议走向‘可委派的异步开发执行单元’。对于工程管理与平台治理，这会带来新的控制面问题：谁能触发 agent、它在哪个环境运行、如何受仓库权限和审计约束，以及哪些自动化动作适合进入受保护分支流程。

Key takeaways:
- PR 评论正在变成 agent 的任务入口，开发协作界面与执行界面开始合并。
- 云端执行环境加上测试回归检查，是 agent 从建议走向落地修改的关键能力。
- 企业需要定义 agent 权限、触发策略和审计要求。

Tags: github-copilot, coding-agent, pull-requests, workflow-automation
Scores: signal 8/5, novelty 7/5, actionability 8/5

## Agent activity in GitHub Issues and Projects
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-03-26
- URL: https://github.blog/changelog/2026-03-26-agent-activity-in-github-issues-and-projects

GitHub 在 Issues 和 Projects 中直接展示 coding agent 的 session 状态，包括 queued、working、waiting for review、completed，并允许从项目视图跳转到 session logs。该能力把 agent 工作流纳入任务计划与执行追踪的统一界面。

Why it matters: 这是组织采用层面的关键信号：agent 不再只是个人工具，而是项目系统中的可观察工作实体。对于工程负责人，这意味着可以开始思考 agent WIP、队列状态、review handoff、失败重试和吞吐量等管理指标，以及如何把 agent 纳入现有 delivery governance。

Key takeaways:
- Agent session 正在被产品化为可管理、可追踪的工作项，而非隐形后台动作。
- 项目工具将逐步需要同时管理人类和 agent 的分工与状态。
- session logs 将成为后续审计、复盘和评估的重要数据源。

Tags: organizational-adoption, agent-ops, github-projects, observability
Scores: signal 7/5, novelty 8/5, actionability 7/5

## View Agentic Workflow configs in the Actions run summary
- Source: GitHub Changelog
- Author: N/A
- Publish date: 2026-03-26
- URL: https://github.blog/changelog/2026-03-26-view-agentic-workflow-configs-in-the-actions-run-summary

GitHub 现在允许在 Actions run summary 中直接查看 Agentic Workflow 的 markdown 配置，减少页面跳转，并显式展示某次运行使用的精确配置。

Why it matters: 虽然是小更新，但对 harness 设计与生产治理很关键。agent 系统的核心问题之一是‘这次到底按什么提示/规则/步骤执行的’；把配置固化到运行摘要中，等于提高了可复现性、审计性与事后归因能力。

Key takeaways:
- Agent workflow 配置需要像 CI 配置一样成为可追踪工件。
- 运行摘要中暴露配置可降低排障和复盘成本。
- 这类细节是把 agent 自动化纳入合规与治理框架的基础。

Tags: agentic-workflows, actions, auditability, harness-design
Scores: signal 6/5, novelty 6/5, actionability 8/5
