## Quantifying infrastructure noise in agentic coding evals
- Source: Anthropic Engineering
- Author: N/A
- Publish date: N/A
- URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic 通过在 Terminal-Bench 2.0 上的内部实验指出，agentic coding eval 的结果会被运行时基础设施配置显著影响：不同资源预算与执行环境之间可产生约 6 个百分点的分差，甚至超过模型排行榜之间的常见差距。文章强调，这类评测不是静态打分，而是把运行环境本身变成了解题过程的一部分，因此不同 CPU、RAM、时间限制和资源执行策略，实质上让模型参加了不同的“考试”。

Why it matters: 这是少见直接挑战 AI 编码榜单可比性的原始技术文章。对于用 SWE-bench、Terminal-Bench 或内部 harness 做模型选型的团队，它提醒你：如果没有把资源配额、超时策略、容器调度与错误归因标准化，评测结果可能更多是在测基础设施而不是测模型。

Key takeaways:
- 在 agentic coding eval 中，运行时环境不是中立容器，而是能力测试的一部分；资源差异会显著改变得分。
- 即使 benchmark 提供推荐 CPU/RAM，如果执行层没有一致、可审计地强制这些约束，结果仍然不可比。
- 对内部 eval 平台而言，应该把 infra error、资源配额、时间预算和重试策略纳入报告，而不是只看最终 pass rate。

## What it actually takes to run code intelligence in-house
- Source: Sourcegraph Blog
- Author: Stephanie Jarmak
- Publish date: 2026-04-21T00:00:00+00:00
- URL: https://sourcegraph.com/blog/what-it-actually-takes-to-run-code-intelligence-in-house

Sourcegraph 从“如果 2026 年从零开始重建一个 Sourcegraph 级别平台，需要什么”这一问题出发，把自建代码智能平台拆成 10 个类别、90 项工程要求，并尝试估算不同环境规模下的三年成本。文章强调，大家通常只看到搜索框与代码浏览器，但真正困难且与 AI coding agent 最相关的部分在于大规模索引、低延迟检索、默认分支与全量历史的不同搜索路径、权限模型、多语言语义索引、以及持续运营。

Why it matters: 虽然这是厂商视角内容，但它提供了一个相对结构化的 buy-vs-build 框架，特别适合正在建设内部代码检索、repo understanding 或 agent context 平台的团队。它的价值不在于结论本身，而在于把“代码智能”从一个功能点还原成一个平台能力集合，帮助工程负责人更现实地评估人力、SRE、权限治理和演进成本。

Key takeaways:
- 对 AI coding agent 而言，代码智能的关键不只是搜索，而是可扩展索引、权限过滤、语义理解与稳定运营的组合能力。
- 开源组件可降低起步门槛，但距离“可在企业内长期运行”的平台仍有显著工程鸿沟。
- 做 buy-vs-build 决策时，应把三年期的人力、基础设施、可靠性和合规成本纳入，而不是只比较原型可行性。

## Implications

- 如果你在做内部 benchmark，必须把机器规格、调度策略、超时、重试、缓存和 infra failure taxonomy 固化到 harness 里，否则结论不稳。
- 如果你在建设 coding agent 平台，检索、索引、权限和上下文供给层的成熟度，可能比继续微调 prompt 更影响最终效果。
- 对企业团队而言，‘能否自建’不再是主要问题，‘能否长期稳定、安全、低延迟地运营’才是主要问题。

## What to Ignore

- 不要把 agentic coding leaderboard 上的细小分差直接解释为模型能力差距，尤其是在基础设施细节未披露时。
- 不要把‘开源组件可用’等同于‘企业级代码智能平台已经基本完成’。
- 不要只看 demo 或 UI 体验，而忽视权限、索引更新、容量规划与故障处理等长期成本。
