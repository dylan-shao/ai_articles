## Quantifying infrastructure noise in agentic coding evals
- Source: Anthropic Engineering
- Author: N/A
- Publish date: N/A
- URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic reports that agentic coding evals are highly sensitive to runtime infrastructure. In internal Terminal-Bench 2.0 experiments, the difference between the most- and least-resourced setups was about 6 percentage points, large enough to exceed typical leaderboard gaps between top models. The core argument is that these are not static benchmarks: because models write code, run tests, install dependencies, and iterate in an environment, the runtime becomes part of the task itself, meaning different resource budgets and enforcement strategies effectively create different tests.

Why it matters: This is a rare primary-source technical post that directly questions the comparability of AI coding leaderboards. For teams using SWE-bench, Terminal-Bench, or internal harnesses to choose models, the message is clear: unless you standardize resource budgets, timeout policy, container scheduling, and infra-failure attribution, your eval may be measuring infrastructure more than model capability.

Key takeaways:
- In agentic coding evals, the runtime is not a neutral container; it is part of the capability test, and resource differences can materially shift scores.
- Even when a benchmark specifies recommended CPU/RAM, results remain non-comparable unless those constraints are enforced consistently and auditably.
- Internal eval platforms should report infra errors, resource budgets, time limits, and retry policy alongside pass rates rather than treating benchmark score as a standalone metric.

## What it actually takes to run code intelligence in-house
- Source: Sourcegraph Blog
- Author: Stephanie Jarmak
- Publish date: 2026-04-21T00:00:00+00:00
- URL: https://sourcegraph.com/blog/what-it-actually-takes-to-run-code-intelligence-in-house

Sourcegraph asks what it would take to rebuild a Sourcegraph-equivalent platform from scratch in 2026, mapping the work to 90 engineering requirements across 10 categories and modeling three-year costs at different scales. The post argues that the visible UI is only the tip of the iceberg; the hard parts, especially for AI coding-agent workflows, are large-scale indexing, low-latency retrieval, dual-path search across default branches versus broader history, permissions, semantic indexing, and long-term operations.

Why it matters: Although this comes from a vendor, it offers a useful and fairly structured buy-vs-build framework for teams considering internal code search, repo-understanding, or agent-context infrastructure. Its value is less the conclusion than the decomposition: it reframes 'code intelligence' from a single feature into a platform capability set, helping engineering leaders estimate staffing, SRE overhead, permission governance, and long-tail maintenance costs more realistically.

Key takeaways:
- For AI coding agents, code intelligence is not just search; it is the combination of scalable indexing, permission-aware retrieval, semantic understanding, and reliable operations.
- Open-source components can lower the initial barrier, but there is still a substantial engineering gap between a prototype and an enterprise-grade long-lived platform.
- Buy-vs-build decisions should include three-year staffing, infrastructure, reliability, and compliance costs rather than comparing only prototype feasibility.

## Implications

- If you run internal benchmarks, machine specs, scheduling policy, timeouts, retries, caching, and infra-failure taxonomy need to be encoded into the harness or your conclusions will be unstable.
- If you are building a coding-agent platform, the maturity of retrieval, indexing, permissions, and context-supply layers may matter more than additional prompt tweaking.
- For enterprise teams, the main question is no longer whether they can build it, but whether they can operate it safely, reliably, and with low latency over time.

## What to Ignore

- Do not treat small leaderboard deltas in agentic coding benchmarks as pure model capability differences, especially when infrastructure details are undisclosed.
- Do not equate 'open-source components exist' with 'an enterprise-grade code intelligence platform is basically solved.'
- Do not over-index on demos or UI while ignoring permissions, index freshness, capacity planning, and failure handling.
