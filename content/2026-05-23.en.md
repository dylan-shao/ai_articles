## Staged publishing and new install-time controls for npm
- Source: GitHub Changelog
- Author: Allison
- Publish date: 2026-05-22
- URL: https://github.blog/changelog/2026-05-22-staged-publishing-and-new-install-time-controls-for-npm/

GitHub announced general availability of npm staged publishing: package tarballs are first uploaded into a staging queue and become installable only after explicit maintainer approval with a 2FA challenge. npm CLI 11.15.0 also adds --allow-file, --allow-remote, and --allow-directory flags, complementing --allow-git for tighter control over accepted install sources.

Why it matters: For platform teams, this upgrades publishing from pure CI credential control to a two-layer model: machine-generated publish intent plus human presence at release time. That is especially useful against compromised pipelines, abuse of trusted publishing with OIDC, and large-scale dependency or release actions triggered by AI or automation agents. The install-source allowlists also provide a practical mechanism to tighten dependency resolution in build environments.

Key takeaways:
- Staged publishing inserts a post-upload approval step backed by 2FA, reducing the chance that non-interactive CI/CD can immediately push a malicious version live.
- The new --allow-* flags make npm install source policy more granular, which is useful for enterprise build baselines and controlled dev containers.

## New Terms

- Staged publishing: an npm release mode where artifacts are uploaded to a staging queue and only become publicly installable after maintainer approval.
- Install source allow flags: npm policy controls such as --allow-file, --allow-remote, --allow-directory, and --allow-git that restrict acceptable install sources.

## Themes

## Implications

- Teams using AI agents, dependency bots, or fully automated release pipelines should evaluate replacing npm publish with npm stage publish and wiring approval into change management.
- Platform teams can begin defining npm install source allowlists for CI, build images, and dev containers to reduce risk from file, remote, or directory-based package inputs.
