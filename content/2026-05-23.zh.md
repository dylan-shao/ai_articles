## Staged publishing and new install-time controls for npm
- Source: GitHub Changelog
- Author: Allison
- Publish date: 2026-05-22
- URL: https://github.blog/changelog/2026-05-22-staged-publishing-and-new-install-time-controls-for-npm/

GitHub 宣布 npm staged publishing 正式可用：包先上传到 stage 队列，需维护者完成显式批准和 2FA 挑战后才会对外可安装。同时，npm CLI 11.15.0 新增 --allow-file、--allow-remote、--allow-directory 安装来源标志，与既有 --allow-git 一起让团队更精细地限制安装时可接受的来源。

Why it matters: 对平台团队而言，这是把“发布权限”从纯 CI 凭证控制提升为“机器提交 + 人工到场批准”的双层机制，尤其适合防御被劫持的流水线、OIDC trusted publishing 滥用、以及 AI/自动化代理发起的大规模依赖与发版操作。安装来源白名单也为构建环境收紧依赖解析路径提供了更现实的执行手段。

Key takeaways:
- staged publishing 要求发布后再由维护者审批，且审批需要 2FA，可显著降低非交互式 CI/CD 直接把恶意版本推到 registry 的风险。
- 新的 --allow-* 标志让组织可以更细粒度地限制 npm install 可接受的来源，适合纳入企业构建基线与受控开发容器。

## New Terms

- Staged publishing：先上传到暂存队列、经维护者审批后才正式发布的 npm 发布模式。
- Install source allow flags：通过 --allow-file/--allow-remote/--allow-directory/--allow-git 限制 npm 安装可接受来源的策略控制。

## Implications

- 若团队已使用 AI 代理、自动依赖升级机器人或全自动发版，应评估是否把 npm publish 切换为 npm stage publish，并把审批纳入变更管理流程。
- 平台团队可以开始为 CI、构建镜像和开发容器定义 npm install 来源白名单，减少来自 file、remote 或目录引用的意外风险。
