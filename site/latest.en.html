Quantifying infrastructure noise in agentic coding evals

  • Source: Anthropic Engineering
  • Author: N/A
  • Publish date: N/A
  • URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic argues that agentic coding benchmarks differ fundamentally from static benchmarks because the runtime environment is part of the task. In internal experiments on Terminal-Bench 2.0, infrastructure resources and execution configuration alone caused a 6-point score swing with statistical significance (p < 0.01).

Why it matters: This is one of the most important evaluation-methodology signals right now. Many teams are using SWE-bench- and Terminal-Bench-style results to drive model selection, budget, and integration decisions; if infra setup alone can exceed leaderboard gaps, then benchmark harnesses, resource budgets, timeouts, and isolation policies must be treated as first-class variables rather than invisible defaults.

Key takeaways: - Agentic coding evals measure not just the model, but also the runtime environment, resource limits, and execution policy. - Without strict control and logging of CPU/RAM/timeout/enforcement methodology, cross-team and cross-vendor scores are not directly comparable. - Benchmark harnesses need production-grade reproducibility, observability, and auditability.

Detecting supply chain attacks at scale with Deep Search

  • Source: Sourcegraph Blog
  • Author: Stephanie Jarmak, Travis Lyons
  • Publish date: 2026-03-26
  • URL: https://sourcegraph.com/blog/detecting-supply-chain-attacks-at-scale-with-deep-search

Using the poisoned LiteLLM releases on PyPI as a case study, Sourcegraph shows how Deep Search and Code Search can quickly identify which public repositories are protected by strict version pinning and which are exposed through unpinned or range-based dependencies, with a reproducible investigation workflow.

Why it matters: This is more than a product post; it is a realistic repo-understanding and code-search operations scenario. For platform teams, the immediate value of AI-assisted search is not just asking questions about code, but doing cross-repo exposure analysis during supply-chain incidents, generating remediation lists, and shrinking time-to-response.

Key takeaways: - The difference between strict pinning and range-based dependency specs directly determines exposure during incidents. - Cross-repo semantic/code search is useful for rapid triage, not just SBOMs or per-repo grep. - Reproducible query and export workflows should become part of incident response playbooks.

Ask @copilot to resolve merge conflicts on pull requests

  • Source: GitHub Changelog
  • Author: N/A
  • Publish date: 2026-03-26
  • URL: https://github.blog/changelog/2026-03-26-ask-copilot-to-resolve-merge-conflicts-on-pull-requests

GitHub Copilot coding agent can now resolve PR merge conflicts by mentioning @copilot in a comment. The agent operates in its own cloud development environment, makes the changes, runs builds and tests, and pushes the result back; the same mechanism also supports fixing failing workflows, addressing review comments, and adding tests.

Why it matters: This shows the shift from IDE-side suggestion tools to delegable asynchronous execution units. For engineering management and platform governance, that creates a new control plane: who can invoke the agent, where it runs, how it is constrained by repo permissions and auditability, and which automated actions are safe to introduce into protected-branch workflows.

Key takeaways: - PR comments are becoming a task-ingress surface for agents, merging collaboration UI with execution UI. - A cloud execution environment plus build/test validation is key to moving from suggestion to committed change. - Enterprises need explicit policies for agent permissions, invocation rules, and auditing.

Agent activity in GitHub Issues and Projects

  • Source: GitHub Changelog
  • Author: N/A
  • Publish date: 2026-03-26
  • URL: https://github.blog/changelog/2026-03-26-agent-activity-in-github-issues-and-projects

GitHub now surfaces coding-agent session states directly in Issues and Projects, including queued, working, waiting for review, and completed, with links into session logs. This integrates agent workflows into the same planning and execution-tracking surfaces used for human work.

Why it matters: This is an important organizational-adoption signal: agents are no longer just personal tools but observable work entities inside the project system. For engineering leaders, that means you can start thinking in terms of agent WIP, queue state, review handoff, retry/failure modes, and throughput, and how to incorporate agents into existing delivery governance.

Key takeaways: - Agent sessions are being productized as manageable, trackable work items rather than hidden background actions. - Project tooling will increasingly need to manage both human and agent allocation/state. - Session logs are likely to become important inputs for audit, retrospectives, and evaluation.

View Agentic Workflow configs in the Actions run summary

  • URL: https://github.blog/changelog/2026-03-26-view-agentic-workflow-configs-in-the-actions-run-summary

GitHub now shows Agentic Workflow markdown configs directly in the Actions run summary, reducing context switching and exposing the exact configuration used for a given run.

Why it matters: While small, this matters for harness design and production governance. Key takeaways: - Agent workflow configuration needs to become a traceable artifact, like CI configuration. - Showing config in run summaries reduces debugging and retrospective overhead. - These details are foundational for bringing agent automation under compliance and governance controls.
