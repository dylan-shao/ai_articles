Organization runner controls for Copilot cloud agent

  • Source: GitHub Changelog
  • Author: N/A
  • Publish date: 2026-04-03
  • URL: https://github.blog/changelog/2026-04-03-organization-runner-controls-for-copilot-cloud-agent/

GitHub added organization-level runner controls for Copilot cloud agent. Previously, the agent environment was mainly configured per repository through copilot-setup-steps.yml; now org admins can set a default runner across repositories and optionally lock it so repos cannot override it. The agent still spins up ephemeral development environments via GitHub Actions, but teams can now standardize on hosted, larger, or self-hosted runners centrally.

Why it matters: This is a foundational step in making agents governable like CI infrastructure. Runner choice determines performance, cost, network reachability, credential boundaries, and access to internal systems. Moving configuration from repo scope to org scope lets platform teams manage the agent execution plane the same way they manage CI, reducing fragmentation and security drift.

Key takeaways: - Copilot cloud agent environments can now be standardized with an org-wide default runner instead of per-repo setup. - Organizations can lock runner policy so the agent always runs on the intended GitHub-hosted, large, or self-hosted execution environment.

Organization firewall settings for Copilot cloud agent

  • Source: GitHub Changelog
  • Author: N/A
  • Publish date: 2026-04-03
  • URL: https://github.blog/changelog/2026-04-03-organization-firewall-settings-for-copilot-cloud-agent/

GitHub added organization-wide firewall controls for Copilot cloud agent. The agent includes a built-in firewall to restrict internet access and mitigate prompt injection and data exfiltration risks. Org admins can now centrally decide whether the firewall is enabled, whether the recommended allowlist is used, maintain an org-wide custom allowlist, and control whether repo admins may add their own allowlist entries.

Why it matters: This addresses one of the main blockers to production agent adoption: once an agent can browse, fetch dependencies, and access external resources autonomously, it needs explicit network boundaries and sane default deny/allow policies. Elevating this to the org layer shows GitHub is treating coding agents as governable execution entities, not just IDE assistants.

Key takeaways: - Org admins can centrally govern whether the agent has internet access, which allowlists apply, and whether repos may extend those rules. - The feature is explicitly aimed at reducing prompt injection and data exfiltration risk while enabling broader enterprise rollout.

Tags: Copilot cloud agent, agent firewall, prompt injection, security governance Scores: signal 9/5, novelty 8/5, actionability 9/5

Copilot cloud agent signs its commits

  • Source: GitHub Changelog
  • Author: N/A
  • Publish date: 2026-04-03
  • URL: https://github.blog/changelog/2026-04-03-copilot-cloud-agent-signs-its-commits/

Copilot cloud agent now signs every commit it creates, causing those commits to appear as Verified on GitHub. This means the agent can now operate in repositories that enforce the 'Require signed commits' branch protection rule, which previously blocked agent usage entirely.

Why it matters: This looks small but is operationally critical. Many regulated or security-conscious teams already require signed commits by policy; if agents cannot satisfy that baseline, they are excluded from serious repositories. Commit signing brings agent-authored changes into existing trust, audit, and supply-chain integrity workflows.

Key takeaways: - Agent-authored commits now carry verifiable identity and integrate with GitHub's Verified commit workflow. - Repositories enforcing signed-commit protection no longer automatically exclude Copilot cloud agent.
