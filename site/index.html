Create issues from Slack with Copilot

  • Source: GitHub Changelog
  • Author: N/A
  • Publish date: 2026-03-30
  • URL: https://github.blog/changelog/2026-03-30-create-issues-from-slack-with-copilot/

GitHub 宣布 Slack 集成新增 Copilot 驱动的 Issue 创建能力：团队成员可在 Slack 频道中 @GitHub，用自然语言直接生成结构化 GitHub Issues，并支持 assignee、labels、milestones 以及父子 issue 层级。用户还可在 Slack 线程中继续迭代需求细节，并通过频道级设置指定默认仓库。

Why it matters: 对平台和工程负责人而言，这代表 AI coding agent 正在从“写代码助手”演进为“团队工作流入口代理”。需求捕获不再局限于开发者手工把聊天内容搬运到 issue tracker，而是直接在协作上下文中生成可执行工单，这会影响需求吞吐、任务结构化质量、项目可追踪性，以及后续 agent 自动接单/生成 PR 的闭环设计。

Key takeaways: - 自然语言到结构化 issue 的转换被前移到 Slack，对跨职能团队更友好。 - 支持子任务层级与线程内迭代，说明 GitHub 正把 agent 能力嵌入项目管理而非只停留在代码编辑层。 - 频道级默认仓库设置降低了多仓库组织中的操作摩擦，是实际落地而非演示型功能。

New Terms

  • Conversation mode：在 Slack 线程中与 @GitHub 往返迭代 issue 内容后再落库。
  • Channel-level instructions：为某个 Slack 频道设置默认工作仓库，影响后续 issue/PR 创建目标。

Themes

  • AI coding 从 IDE 内助手扩展到聊天与项目管理入口。
  • 工单 intake 自动化成为 agent 工作流的重要前置层。
  • 开发工具竞争正在转向跨系统编排，而不只是代码补全质量。

Implications

  • 组织应开始定义‘聊天到工单’的质量门槛，例如标签策略、模板规范、默认仓库映射和审计要求。
  • 如果后续再接上 agent 自动实现 issue、生成 PR、更新状态，Slack 可能成为开发工作流的新控制面。
  • 平台团队需要考虑权限边界：谁能在聊天里触发 issue 创建、写入哪些仓库、是否需要审批与可追踪日志。
