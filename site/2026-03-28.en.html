Quantifying infrastructure noise in agentic coding evals

  • Source: Anthropic Engineering
  • Author: N/A
  • Publish date: N/A
  • URL: https://www.anthropic.com/engineering/infrastructure-noise

Anthropic examines a frequently overlooked variable in agentic coding benchmarks: infrastructure configuration. In internal Terminal-Bench 2.0 experiments, different resource setups and enforcement methods produced roughly a 6-point score swing, larger than the margin between many frontier models on public leaderboards. Because these evaluations let models write code, run tests, install dependencies, and iterate in a live environment, the runtime is part of the task itself rather than a neutral container; CPU, RAM, time limits, and enforcement mechanics materially change what is being measured.

Why it matters: This piece challenges the default assumption that coding benchmark scores are clean measurements of model quality. For teams comparing models, agents, or internal stacks, any conclusion without a standardized harness, resource budget, and failure attribution policy may mostly reflect who benefits from a particular environment rather than who is genuinely better at software engineering tasks.

Key takeaways: - Unlike static benchmarks, agentic coding evals make the runtime environment part of the task, so infra is a first-class experimental variable. - Resource budgets and enforcement methodology can materially shift scores; without standardization, small model deltas are not trustworthy. - Eval design should specify CPU, RAM, timeout, retry policy, container isolation, and infra error attribution, or leaderboard-driven decisions may be misleading.

Themes

  • Benchmark comparability is increasingly constrained by harness standardization and resource governance, not just prompts and models.
  • Engineering organizations should treat eval platforms as production-grade infrastructure.

Implications

  • If you run internal coding-agent bake-offs, freeze container images, CPU/RAM quotas, timeout and retry rules, and separately track infra failures.
  • If you rely on public leaderboards for model selection, ask for runtime and enforcement details because score differences may not be operationally meaningful.
  • If you design your own benchmark, make resource specs and enforcement mechanics part of the protocol and provide reproducible harness scripts.
